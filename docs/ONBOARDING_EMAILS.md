# DanteForge Onboarding Emails

## Day 0

**Subject:** You do not need a bigger Claude session. You need a tighter workflow.

```text
Hey,

You are in.

DanteForge exists for one reason: Claude gets expensive when every session has to carry the whole project in its head.

The fix is not "prompt harder."

The fix is running the work in smaller, cleaner waves.

Start here:

1. Install DanteForge
2. Run `danteforge init`
3. Run `danteforge review`
4. Run `danteforge magic "your actual goal"`

What you should feel after the first pass:

- the repo is grounded
- the next step is obvious
- the model is working on the current slice, not reconstructing your whole project from scratch

If you only do one thing today, do this:

Run `danteforge review` inside a repo you know well.

You will immediately see where the wasted context usually starts.

More soon,

Richard
Founder, DanteForge
```

## Day 3

**Subject:** The feature most people miss: local-first planning before the expensive model step

```text
Hey,

Most people use DanteForge once, see the workflow, and then miss the real power move.

The power move is this:

Do the planning first.
Do as much of it locally as possible.
Only spend premium model time when the task is already narrow.

That means:

- review
- specify
- clarify
- plan
- tasks

before you ask Claude to do the expensive part.

This sounds small. It is not.

It is the difference between:

"Here is my whole repo, all my constraints, and everything I have tried"

and:

"Implement task 3 from this plan and keep the change inside these files."

That is where the session savings come from.

If you want the shortest path to feeling this in practice:

1. Run `danteforge spark "your next feature"`
2. Read the plan
3. Then run `danteforge magic "finish task 1"`

Same project.
Much smaller context.
Much better output.

Richard
```
